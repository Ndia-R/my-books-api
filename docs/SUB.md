# 実装依頼：サブスクリプションプラン（subscriptionPlan）に基づく認可ロジックの実装

## 1. 目的

本プロジェクトでは、Keycloak 由来の **Role（役割）** と、アプリ DB 内で管理する **subscriptionPlan（契約状態）** の二段階で認可を制御します。既に DB へのカラム追加およびユーザー情報取得の基盤は完了しているため、これを利用して「全文閲覧」等のビジネスルールに基づくアクセス制御ロジックの実装をお願いします。

## 2. 認可モデルの再確認

本システムの認可は以下の 2 層構造で判定します。

1. **Role（権限レイヤー）**: 「その操作を行う資格があるか」を判定。
* 例：`book-content:read:any` という Permission を持っているか。


2. **subscriptionPlan（属性レイヤー）**: 「プランの内容に応じてどこまで許容するか」を判定。
* 例：`FREE` プランなら試し読みのみ、`PREMIUM` プランなら全文閲覧を許可。



> **重要：** BFF がアプリ DB（subscriptionPlan）に依存することを避けるため、BFF は JWT の中継に徹し、具体的なプラン判定はリソースサーバー（本 API）内で行います。

## 3. 実装対象および仕様

### 3-1. 書籍全文閲覧ロジックの構築

書籍詳細取得 API において、以下のロジックを実装してください。

| プラン | 閲覧範囲 | 期待される挙動 |
| --- | --- | --- |
| **FREE (一般)** | 試し読み分のみ | 全文データをマスク、またはサマリーのみを返却 |
| **PREMIUM (有料)** | 全文 | 全文データを返却 |

* **実装箇所**: `BookService` 等のビジネスロジック層。
* **判定方法**: `SecurityContext` から取得した `userId` を元に、DB から最新の `subscriptionPlan` を取得して判定してください。

### 3-2. Permission との組み合わせ判定

単にプランを見るだけでなく、`ROLE_USER` などの役割を持っていることが前提となります。

* `@PreAuthorize("hasAuthority('book-content:read:any')")` で機能自体のアクセス権を確認した上で、メソッド内でプランに応じたデータフィルタリングを行ってください。

### 3-3. パフォーマンスへの配慮

API リクエストのたびにプラン確認の DB クエリが発生するため、以下の検討をお願いします。

* **キャッシュの導入**: `subscriptionPlan` は頻繁に変更されないため、Redis または Spring Cache (`@Cacheable`) を利用した一時的なキャッシュ。
* **一貫性**: プランが更新された際のキャッシュクリア戦略。

## 4. 慎重に検討してほしいポイント

* **未ログインユーザーの扱い**: `permitAll()` で公開されている API（書籍検索等）において、認証がない場合に `subscriptionPlan` 判定をどうバイパスし、デフォルトの `FREE` 相当の挙動をさせるか。
* **エラーハンドリング**: 権限はあるがプランが不足している場合（例：一般会員が全文 API を叩いた場合）は、単なる `403 Forbidden` ではなく、フロントエンドが「アップグレード案内」を出せるような独自のエラーレスポンスを検討してください。
